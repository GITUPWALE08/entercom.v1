# Frontend Response Normalization & Runtime Stability Audit

You are a Senior React + TypeScript Architect working on the Entercom frontend.

Your objective is to eliminate all runtime crashes caused by inconsistent backend response shapes and make the frontend resilient to future backend changes.

## Background

The backend intentionally returns two response patterns.

### Pattern A (DRF Pagination)

```json
{
  "count": 0,
  "next": null,
  "previous": null,
  "results": []
}
```

### Pattern B (Application Envelope)

```json
{
  "success": true,
  "message": "...",
  "data": [...],
  "pagination": {
    "count": 10,
    "next": null,
    "previous": null
  }
}
```

The frontend currently assumes every endpoint returns an array directly.

This causes runtime errors such as:

```text
TypeError:
(data || []).filter is not a function
```

because data is actually an object.

---

# PRIMARY GOAL

Do NOT patch individual pages.

Fix the architecture.

Implement a centralized response normalization layer.

Every React component should receive predictable data structures regardless of backend response format.

---

# REQUIRED TASKS

## Phase 1 — Audit

Search the entire frontend for:

* .filter(
* .map(
* .reduce(
* .find(
* .some(
* .every(
* .length
* Array.isArray
* response.data
* query.data

Locate every place where array methods are called.

Determine whether the variable may actually contain:

* Application envelope
* DRF pagination object
* plain array
* undefined
* null

Generate an audit report.

---

## Phase 2 — Normalize API Layer

Locate all API client/service files.

Examples:

* api.ts
* axios.ts
* requestService.ts
* notificationService.ts
* hooks
* React Query fetchers

Create ONE reusable normalization utility.

Example behavior:

Application Envelope

↓

returns

```ts
response.data
```

DRF Pagination

↓

returns

```ts
response.results
```

Plain Array

↓

returns itself

Object

↓

returns object

The rest of the application should never know which backend format was received.

---

## Phase 3 — Refactor Components

Update every page, hook and component to consume normalized responses instead of raw axios responses.

Remove duplicated extraction logic such as:

```ts
response.data.data
```

or

```ts
response.results
```

Components should only consume already-normalized values.

---

## Phase 4 — Defensive Programming

Where arrays are expected:

Never assume.

Use:

* Array.isArray()
* null coalescing
* optional chaining

Prevent crashes from:

undefined

null

objects

unexpected payloads

---

## Phase 5 — React Query

Audit every query.

Ensure:

query.data

always has a deterministic shape.

Avoid:

```ts
query.data.filter(...)
```

unless query.data is guaranteed to be an array.
