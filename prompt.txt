## GLOBAL LOADING & PRELOADER UX STANDARDS

The Entercom platform is already in production with real users.

The objective is to improve perceived performance and user experience, **not** to add loading animations everywhere.

The application must intelligently distinguish between:

* Application Preloader
* Page Loader
* Component Loader
* Button Loader
* Skeleton Loader

Never use them interchangeably.

---

# 1. APPLICATION PRELOADER

Use the official Entercom company logo as the application preloader.

Purpose:

Provide visual feedback while the application is performing heavy initialization.

The logo animation should feel premium, smooth, and lightweight.

Do NOT create an unnecessarily long splash screen.

---

## When the Application Preloader SHOULD appear

Only display the preloader when one or more of these are true:

* First application launch
* Hard refresh
* Initial authentication bootstrap
* User profile initialization
* Permission/RBAC initialization
* Large dashboard initialization
* Initial React Query cache hydration
* Heavy application initialization (>2 seconds)

Typical examples:

* Login → Customer Portal
* Login → Manager Portal
* Login → Staff Portal
* Login → Technician Portal
* Login → Admin Portal

---

## When it MUST NOT appear

Never show the preloader for:

* Route navigation between already loaded pages
* Opening a modal
* Opening a drawer
* Clicking sidebar links
* Viewing request details
* Viewing product details
* Small API refreshes
* Quick state updates

---

## Timing Rules

Average load <2 seconds

→ Do NOT show the preloader.

Average load between 2–3 seconds

→ Show the logo preloader.

Longer than 3 seconds

→ Continue showing the logo but also display progress feedback such as:

* "Loading dashboard..."
* "Preparing workspace..."
* "Fetching your requests..."

Never block the user for more than **5 seconds** without meaningful feedback.

---

# 2. PAGE LOADERS

Use page-level loaders only when an entire page is waiting for its primary data.

Examples:

* Dashboard
* Requests page
* Orders page
* Inventory
* Analytics
* Chat Inbox

Preferred approach:

* Skeleton UI
* Partial placeholders
* Progressive rendering

Avoid blank white screens.

---

# 3. COMPONENT LOADERS

Use localized loaders only.

Examples:

* Charts
* Tables
* Notification list
* Product cards
* Conversation list

Only the affected component should show loading.

The rest of the page must remain interactive.

---

# 4. BUTTON LOADERS

Buttons performing async actions must show inline loading.

Examples:

* Login
* Register
* Submit Request
* Approve Quote
* Reject Quote
* Save
* Delete
* Upload
* Checkout
* Assign Technician
* Resolve Conversation

Requirements:

* Spinner inside the button
* Disable repeated clicks
* Preserve button width
* Do not shift layout

---

# 5. FORM LOADERS

Large forms should never trigger a full-page loader.

Instead:

* Disable the form
* Show loading only on the submit button
* Keep entered values visible

---

# 6. TABLE LOADERS

Tables should use skeleton rows instead of generic spinners.

Maintain:

* Table headers
* Pagination
* Layout

Only rows should be skeletonized.

---

# 7. CARD LOADERS

Dashboard KPI cards should use skeleton cards.

Do not replace the entire dashboard with a spinner.

---

# 8. IMAGE LOADERS

Images should:

* Lazy load
* Fade in
* Show placeholders while loading

Broken images should gracefully fall back to a default placeholder.

---

# 9. SEARCH LOADERS

Searching should not trigger full-page loading.

Only update:

* Search results
* Result count

Keep filters and search inputs interactive.

---

# 10. WEBSOCKET UPDATES

Never show loaders for:

* Real-time notifications
* Incoming chat messages
* Live request updates
* Dashboard counters

Use subtle animations instead.

---

# 11. PAGINATION

Changing pages should:

* Keep existing content visible
* Show skeletons only in the incoming section
* Avoid flashing the entire screen

---

# 12. LOADING CONSISTENCY

Standardize all loading indicators across the application.

Use one consistent design system for:

* Spinner
* Skeleton
* Progress bar
* Fade animation
* Logo preloader

---

# PERFORMANCE RULES

Avoid loaders for operations completing in under **300ms** to prevent distracting flicker.

Prefer optimistic UI updates where safe.

Reuse cached React Query data while refetching in the background.

---

# UX AUDIT

Audit every page and determine the correct loading strategy.

Replace any inappropriate full-page spinners with:

* Skeleton loaders
* Inline loaders
* Button loaders
* Component loaders

Use the application logo preloader **only** where it genuinely improves perceived performance.

---

✔ Company logo preloader appears only during heavy application initialization.

✔ No unnecessary splash screens.

✔ No full-page loaders for small interactions.

✔ Buttons use inline loading.

✔ Tables use skeleton rows.

✔ Cards use skeleton placeholders.

✔ Images use placeholders and fade-in loading.

✔ Forms preserve entered data while submitting.

✔ WebSocket updates never block the UI.

The final experience should feel fast, modern, and intentional rather than overloaded with animations.
