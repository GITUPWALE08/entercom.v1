# Entercom — Phase 7 Support Chat System (Production-Safe MVP)

You are a Senior Django + React Real-Time Systems Architect implementing **Phase 7: Support Chat System** for the Entercom platform.

IMPORTANT CONTEXT:

* The platform is already deployed in production with real users.
* Phases 1–6 are implemented (Auth, RBAC, Requests, Bookings, Payments, Notifications, Celery, Redis, Channels).
* We must avoid architectural drift and avoid introducing unnecessary complexity.
* Existing request workflows, payment flows, and notification infrastructure are authoritative and must not be broken.

GOAL:
Implement a **production-safe MVP support chat system** that allows customers, staff, managers, and technicians to communicate in real time about requests, bookings, payments, and general support issues.

This is NOT a social chat app. It is an operational support messaging system.

==================================================
PHASE 7 MVP SCOPE (ONLY THESE FEATURES)
=======================================

Implement exactly:

1. Conversation creation
2. Conversation participants
3. Message sending
4. Message retrieval (paginated)
5. Real-time message delivery via WebSockets
6. Read receipts (conversation-level, not per-message reactions)
7. Unread count
8. Request-linked conversations
9. Staff assignment to a conversation
10. Audit logging for privileged actions

DO NOT implement:

* Voice/video calls
* Typing indicators
* Emoji reactions
* File attachments (defer)
* AI auto-replies (Phase 8)
* Push notifications beyond existing notification infrastructure
* WhatsApp integration
* Multi-tenant organizations
* End-to-end encryption

==================================================
RBAC RULES
==========

Customer:

* Can create support conversations.
* Can view only conversations they participate in.
* Can send messages only in their conversations.
* Cannot assign staff or close conversations.

Technician:

* Can view conversations linked to requests assigned to them.
* Can send messages in those conversations.
* Cannot view unrelated customer conversations.

Staff:

* Can view all support conversations.
* Can assign themselves.
* Can resolve conversations.
* Can add participants.

Manager:

* Full chat administrative access.

Superadmin:

* Full access.

==================================================
WEBSOCKET IMPLEMENTATION
========================

Use Django Channels with JWT authentication.

Route:
/ws/chat/{conversation_id}/

Requirements:

* Authenticate using JWT token.
* Verify user is a participant before joining group.
* Reject unauthorized connections with close code 4403.
* Broadcast only new messages and read events.
* No global chat room.
* One Redis group per conversation.

Implement bounded reconnect strategy on the frontend (max 5 attempts, exponential backoff).

==================================================
NOTIFICATION INTEGRATION
========================

When a new message is sent:

* Trigger existing DispatchOrchestrator event: support_message_received
* Notify all participants except sender.
* Use existing notification channels (in-app first; email optional).

Do NOT send email for every message by default. Only create the event hook.
