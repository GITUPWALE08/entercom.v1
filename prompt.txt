# Entercom — Production Gap Hunter (Frontend + Backend)

You are a Senior Production Reliability Engineer and QA Architect auditing the Entercom platform.

## CONTEXT

The platform is already deployed in production and is being tested by real users. We are NOT looking for code style issues or theoretical improvements. We are looking for **hidden production failures** that only appear when real users move through edge-case workflows.

A real bug that was already discovered:

* Customer created a MAINTENANCE request.
* Staff generated a quote.
* Customer attempted payment.
* Frontend redirected to `/orders/:id`.
* Maintenance requests do not create orders.
* `order_id` was null, causing a production crash.

This is the class of bug we want to systematically uncover.

---

# YOUR MISSION

Perform a **Production Gap Hunt** across BOTH frontend and backend.

Do NOT assume workflows are correct. Treat every cross-domain handoff as potentially broken.

---

# AUDIT AREAS

## 1. Authentication & Onboarding

Test all combinations:

* Register → verify email → login
* Register → do not verify → login attempt
* Password reset on verified user
* Password reset on unverified user
* Expired reset token
* Multiple reset requests in parallel
* Login while account is locked
* Refresh token after logout
* Refresh token after password change
* Refresh token after role change

Look for:

* Stale sessions
* Incorrect redirects
* Infinite loading states
* Missing error handling
* Race conditions

---

## 2. Request Lifecycle Chaos Testing

For EACH request category:

* installation
* maintenance
* inspection
* support
* consultation
* warranty
* booking
* product_order
* device_outage

Generate a matrix of:
Request Created → Quote Created → Payment Required? → Order Exists? → Technician Required? → Verification Required? → Completion Path.

Identify every place where the frontend assumes a field exists when it may legally be null.

Examples to check:

* `order_id`
* `payment_id`
* `booking_id`
* `assigned_technician`
* `verification_record`
* `quote`
* `next_states`
* `location.postal_code`
* `customer.phone`

---

## 3. Payment Flow Stress Test

Simulate:

* Payment success
* Payment cancelled by user
* Payment expired
* Webhook arrives before frontend callback
* Frontend callback arrives before webhook
* Duplicate webhook delivery
* Payment for quote without order
* Payment for order without quote
* Payment amount mismatch
* Currency mismatch
* Network drop during redirect

Verify:

* Idempotency
* Correct post-payment redirect
* No orphan payments
* No orphan orders
* No double charging
* UI recovery after refresh

---

## 4. Technician Workflow Gap Hunt

Use a fresh technician account.

Test:

* Assignment appears
* Accept assignment
* Decline assignment
* Accept from two browser tabs simultaneously
* Create quote twice
* Submit verification twice
* Upload evidence with slow network
* Complete job after quote expires
* Attempt completion without verification

Look for:

* Duplicate records
* Missing permissions
* State machine bypasses
* Stuck assignments

---

## 5. Staff & Manager Failure Paths

Test actions that override normal flow:

* Reassign technician
* Cancel assigned request
* Approve verification
* Reject verification
* Force complete request
* Reopen completed request
* Edit quote after customer approval
* Approve expired quote
* Override payment status

Verify backend rejects illegal transitions even if the frontend attempts them.

---

## 6. Frontend Null-Safety Audit

Search the React codebase for patterns that commonly crash in production:

* `.map(`
* `.filter(`
* `.find(`
* `.length`
* `Object.keys(`
* `Object.values(`
* direct property access (`obj.a.b.c`)

For each occurrence, determine whether the API can legally return null/undefined/non-array values.

Flag any unsafe assumption as **P0** if it can crash the UI.

---

## 7. React Query / Cache Invalidation Audit

Verify that after these mutations the correct queries are invalidated:

* Create request
* Update request
* Cancel payment
* Cancel order
* Approve quote
* Assign technician
* Mark notification as read
* Update profile
* Logout

Look for stale UI where the backend succeeded but the frontend still shows old data until manual refresh.

---

## 8. Production-Only Environment Gaps

Audit differences between local and production:

* Render environment variables
* Railway worker variables
* Vercel frontend variables
* Resend API key
* Upstash Redis URL
* HTTPS vs HTTP
* CORS origins
* CSRF trusted origins
* WebSocket URLs
* Build-time vs runtime variables in Vite

Identify any variable that is available locally but not guaranteed in production.

---

## 9. Observability Gaps

For every critical failure path, verify there is enough information to debug production incidents.

Required signals:

* request_id
* user_id
* public_id
* payment_reference
* quote_id
* task_id
* provider_response_id
* status_before
* status_after
* actor_role

Flag any failure that would be impossible to investigate from logs alone.
