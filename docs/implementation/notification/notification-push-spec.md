# Notification Push Specification

## Purpose
The purpose of this document is to define the technical requirements for formatting and dispatching Push Notifications to mobile devices and Progressive Web Apps (PWAs). It addresses the unique constraints of push providers (like Apple APNS and Google FCM), specifically payload size limits and token lifecycle management.

## Scope
- Device token retrieval and validation.
- Constructing the provider-specific JSON payload (APNS/FCM).
- Handling silent pushes vs. visible alerts.
- Managing invalid, expired, or unregistered device tokens.

## Out of Scope
- Mobile client code for registering for push permissions.
- PWA Service Worker implementation.
- Selecting between APNS or FCM (this document assumes a generic push dispatch strategy).

## Definitions
- **Device Token**: A unique string generated by the OS (iOS/Android) and provided to the frontend client, which is then securely stored in the backend Users domain to target that specific device.
- **Silent Push (Data Message)**: A push payload that does not trigger a visual alert or sound on the device but wakes up the background application to fetch new data.
- **Visible Push (Notification Message)**: A push payload that the OS renders directly as a banner, playing a sound or vibrating the device.

## Architecture

### 1. Token Retrieval and Multiplexing
A single user may have multiple registered devices (e.g., a tablet and a smartphone).
- When the `DeliveryProcessingService` executes a Push dispatch task, it must query the Users domain for all active device tokens associated with the `recipient_id`.
- The dispatcher creates a single API request to the Push provider containing the array of tokens, allowing the provider to handle the fan-out to the individual devices.
- If the user has 0 active device tokens, the delivery is immediately marked as a Permanent Failure (`DEAD_LETTERED`) with the reason "No active devices", avoiding unnecessary retries.

### 2. Payload Construction Constraints
Push providers enforce strict limits on payload size (e.g., APNS restricts payloads to 4KB).
- **Core Fields**: The payload MUST strictly contain only the `title` and a heavily truncated `message`.
- **Deep Linking Context**: Crucial routing information (e.g., `resource_type`, `resource_id`) must be included in the custom data dictionary to allow the mobile app to route the user to the correct screen when they tap the notification.
- **Avoid Large Data**: The payload MUST NOT include full object representations (like the entire Request JSON).

### 3. Token Lifecycle Management (Unregistered Devices)
Users frequently uninstall applications or revoke OS-level push permissions.
- **Provider Feedback**: When the Push Dispatcher communicates with the provider API, the provider will return a response indicating if any specific tokens in the batch are now invalid or unregistered.
- **Token Invalidation**: The Dispatcher MUST parse these provider errors. If a token is marked invalid (e.g., FCM `UNREGISTERED` error), the Dispatcher must fire a command to the Users domain to immediately delete or disable that specific device token in the database.
- **Status Mapping**: If *some* tokens succeed but *others* fail, the overall `DeliveryAttempt` is considered `SENT` (success), but the invalid tokens are cleaned up to optimize future dispatches.

## Responsibilities
- **Push Dispatcher**: Construct minimal payloads, respect OS limits, and aggressively parse provider feedback to prune dead tokens.
- **Users Domain**: Maintain a relational table of valid device tokens mapped to user IDs, exposing APIs for the frontend to register tokens and the Notification domain to retrieve/invalidate them.

## Dependencies
- **Users Domain**: The absolute source of truth for Device Tokens.

## Open Questions
- None. The specification clearly outlines the constraints and token management requirements.

## Completion Criteria
- The strategy for handling multiple devices per user is defined.
- Payload size constraints and data mapping rules are codified.
- The critical requirement for automated invalid token pruning is established.
