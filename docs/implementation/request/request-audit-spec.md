# Request Audit Specification

## 1. File Purpose
Defines the mandatory logging requirements for Phase 3. This specification ensures forensic traceability, compliance with financial rules, and accountability for quality overrides.

## 2. Scope
*   Mandatory audit fields and metadata.
*   Specific requirements for financial, quality, and managerial actions.
*   Immutability expectations.

## 3. Out of Scope
*   Technical implementation of the Audit Log microservice/app.
*   Log rotation or archiving policy.

## 4. Full Content

### 4.1 Mandatory Audit Fields
Every audit record generated by the Request system MUST include:
*   `request_id`: Primary Request UUID.
*   `actor_id`: UUID of the User or System Service.
*   `action`: Canonical action name (e.g., `request.quote.approved`).
*   `timestamp`: Precision ISO8601.
*   `correlation_id`: Link to the initiating event/transaction.
*   `previous_state` / `new_state`: String representations of state.
*   `reason`: Mandatory for cancellations, declines, and overrides.
*   `metadata`: JSON object containing context (IP address, Browser, Device Info).

### 4.2 Audit Categories

#### 4.2.1 Financial Audit (High Sensitivity)
*   **Triggers**: Quote creation, revision, approval, rejection.
*   **Requirements**: Must record version number, amount, and initiator role. Revisions must include the "Reason for Revision".

#### 4.2.2 Verification Audit (Forensic)
*   **Triggers**: Evidence submission, QA Pass, QA Fail.
*   **Requirements**: Must record links to all evidence objects. Failures must include the specific QA checklist rejection points.

#### 4.2.3 Managerial Actions (Highest Risk)
*   **Triggers**: Verification Override, Cancellation after Assignment, Manual Escalation.
*   **Requirements**: These actions require a **mandatory reason** and must be flagged for secondary security review in audit dashboards.

#### 4.2.4 Assignment Actions
*   **Triggers**: Assign, Accept, Decline, Timeout.
*   **Requirements**: Declines must log the reason code (e.g., `safety_concern`).

### 4.3 Immutable Audit Expectations
*   Audit logs must be stored in a write-once database (enforced via row-level triggers or application-level guards).
*   Forensic logs must be emitted to the system stream *before* the DB transaction completes (Dual-Write pattern) to prevent loss on rollback.
