# Notification Admin Tools Architecture

## Purpose
The purpose of this document is to define the administrative toolset required to monitor, troubleshoot, and manually govern the Notification Domain. It ensures that system operators have the necessary visibility and control over notification flows without requiring direct database manipulation.

## Scope
- Administrative dashboards and visibility into delivery metrics.
- Tooling for inspecting individual notification delivery states and payloads.
- Mechanisms for manually intervening in failed deliveries (e.g., forced re-dispatch).
- Tools for auditing user preference configurations.

## Out of Scope
- Frontend UI component designs for the admin dashboard.
- Modifying the core business events that trigger the notifications.
- Sending ad-hoc, custom marketing messages (this is a marketing tool, not an administrative debugging tool).

## Definitions
- **Dead-Letter Inspection**: The ability to view the exact API response or error trace that caused a notification delivery to fail permanently.
- **Forced Re-dispatch**: A manual command that overrides a `Dead-Lettered` or `Failed` state, placing the delivery attempt back into the `Pending` queue.

## Architecture

### Diagnostic Visibility
The Admin Tools architecture relies on the Notification Domain exposing specialized read-only endpoints intended strictly for users with the `Admin-User` role.
1. **Global Health Metrics**: Aggregation endpoints providing real-time counts of `Pending`, `Failed`, and `Dead-Lettered` deliveries to alert operators to systemic provider outages (e.g., a massive spike in failed emails).
2. **User Inbox Inspection**: Endpoints that allow an admin to view a specific user's notification history (excluding the ability to mutate the Read/Unread state) to verify if an alert was actually generated by the system.
3. **Preference Inspection**: Endpoints allowing support staff to view a user's current preference overrides (e.g., to confirm the user opted out of emails, explaining why they didn't receive one).

### Intervention Mechanisms
To recover from system failures, the domain exposes strict, auditable mutation endpoints:
1. **Re-dispatch API**: Accepts a specific `Delivery ID`. Validates that the delivery is in a terminal failure state, logs the administrative action to the Audit Domain, and resets the state to `Pending`, allowing the background workers to attempt delivery again.
2. **Bulk Re-dispatch**: An asynchronous task triggered by an admin to retry a large batch of failed deliveries (e.g., after an external provider resolves an outage).

## Responsibilities
- **Admin Service Layer**: Validate administrative authorization before interacting with the core Notification tables.
- **Audit Integration**: Guarantee that every manual intervention (like a forced re-dispatch) is published to the Audit Domain with the Admin's `actor_id`.

## Dependencies
- **Roles/Auth Domain**: For verifying that the user interacting with these tools possesses the necessary operational permissions.
- **Audit Domain**: For recording the administrative actions taken.

## Open Questions
All business and technical decisions have been resolved:

**Question:** Should the system automatically pause all outbound dispatching for a specific channel if the failure rate exceeds a certain threshold, requiring an admin to manually "unpause" it?
**Decision:** Yes.

Implement automatic channel circuit-breaking.

If a notification provider exceeds the configured failure threshold, the channel automatically enters a PAUSED state.

Only Administrators may manually resume the channel after investigation.

Rationale:
This prevents runaway retry storms, protects provider reputation, and avoids unnecessary infrastructure costs.

Impact:
Notification Dispatcher
Delivery Monitor
Admin Notification Console
Notification Failure Service

**Question:** Are admins permitted to manually modify a user's notification preferences on their behalf?
**Decision:** Yes.

Only Administrators may modify another user's notification preferences.

Every modification must require a reason and generate an immutable audit record.

Rationale:
Support staff frequently need to assist customers.

Auditability prevents abuse.

Impact:
Notification Preferences
Audit Logs
RBAC

## Completion Criteria
- The specific capabilities required for system operators to monitor and intervene in the notification flow are established.
- The requirement for strict auditing of these administrative actions is codified.
