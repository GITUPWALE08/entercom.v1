# Payment Service Architecture

## 1. Purpose
The purpose of this document is to define the logical service boundaries and responsibilities for managing Payments, Payment Provider integration, and Webhook processing within the Entercom platform. It establishes the strict security and transactional rules required to ensure that financial state transitions are authoritative, verified, and consistent across all domain aggregates.

## 2. Scope
This document covers:
* **PaymentService:** Management of internal payment records and lifecycle orchestration.
* **ProviderService:** Encapsulation of low-level communication with the Payment Provider API.
* **WebhookService:** Authoritative processing and verification of provider notifications.

## 3. Out of Scope
* Refund processing or reversed transaction logic.
* Multi-currency conversion or settlement (Phase 5 is local currency only).
* Recurring billing or subscription management.
* Frontend SDK integration or client-side callback handling.

## 4. Definitions
* **Authoritative Webhook:** A server-to-server notification from Payment Provider that has been cryptographically verified by the platform.
* **Transactional Idempotency:** The guarantee that processing a redundant webhook event for the same transaction will result in zero duplicate state changes or side effects.
* **Payment Reference:** A unique identifier generated by the system and shared with Payment Provider to track the transaction lifecycle.

## 5. Service Inventory

### 5.1 PaymentService
* **Purpose:** Orchestrates the internal representation and lifecycle of a financial transaction.
* **Responsibilities:**
    * **Initialize Payment:** Create the initial Payment record linked 1:1 to an Order.
    * **Generate References:** Produce unique identifiers for Payment Provider correlation.
    * **Track Payment Lifecycle:** Maintain states from `pending` through terminal `paid` or `cancelled`.
    * **Expire Payments:** Background Job is the sole producer for payment expiry. Expiry is not user initiated.
* **Inputs:** Order ID, Amount, Actor ID.
* **Outputs:** Payment aggregate state, Reference ID.
* **Events Produced:** `payment.initialized`, `payment.expired`.

### 5.2 ProviderService
* **Purpose:** Provides a clean, abstracted interface for interacting with the Payment Provider provider API.
* **Responsibilities:**
    * **Communicate with Payment Provider:** Perform API requests for transaction initialization and verification.
    * **Verify Transactions:** Perform server-to-side verification checks against the Payment Provider API.
    * **Manage References:** Map platform references to provider-specific data.
* **Boundary Rule:** **No business state transitions occur here.** This service is restricted to communication and data translation; it does not update Payment or Order statuses directly.
* **Inputs:** Payment data, Provider Reference.
* **Outputs:** Provider-specific response data, Verification status.
* **Events Produced:** None (Operational only).

### 5.3 WebhookService
* **Purpose:** Acts as the secure, authoritative entry point for provider-initiated state changes.
* **Responsibilities:**
    * **Receive Webhook:** Process the raw HTTP request from Payment Provider.
    * **Verify Signature:** Validate the HMAC-SHA512 header using the shared secret key. Invalid or rejected webhooks result in the Payment transitioning to failed.
    * **Perform Idempotency Check:** Verify that the transaction hasn't already been processed into a terminal state.
    * **Update Payment:** Transition the Payment record to `paid` or `failed`.
    * **Emit Event:** Emit payment.paid event. WebhookService MUST NOT directly mutate Order state or trigger Inventory Reduction.
* **Inputs:** Raw Request Body, HMAC Signature.
* **Outputs:** Success/Failure acknowledgment (200 OK for Payment Provider).
* **Events Produced:** `payment.paid`, `payment.failed`, `webhook.processed`.

## 6. Ownership Boundaries
* **Canonical Truth Rule:** **The Webhook is the sole authoritative source of truth.** No human role and no frontend signal can transition a payment to the `paid` state.
* **Signature Mandate:** Only verified webhooks (passing HMAC validation) are permitted to transition payment states.
* **Expiration Policy:** **Payment expires strictly after 24 hours.** This is a system-enforced invariant managed by PaymentService.

## 7. Event Responsibilities
Services must emit domain events to notify the Order and Inventory domains of settlement, ensuring that stock reduction is atomically coupled with payment success.

## 8. Audit Responsibilities
* **PaymentService:** Log all lifecycle transitions and expiration triggers.
* **WebhookService:** Capture raw payload hashes (no secrets), signature verification results, and idempotency hits for forensic tracing.

## 9. Dependencies
* **Payment Provider API:** External provider for payment processing.
* **Order Domain:** Parent aggregate that provides amount data and consumes status updates.
* **Inventory Domain:** Consumer of payment success signals for stock depletion.

## 10. Open Questions
* **UNRESOLVED — BUSINESS DECISION REQUIRED:** Does WebhookService require a secondary verification call back to ProviderService (polling) after receiving a webhook, or is the HMAC signature check considered sufficient for settlement?

## 11. Completion Criteria
* Service boundaries prevent any manual override of payment success.
* Transactional safety ensures inventory is only reduced when payment is 100% verified.
* Payment Provider-specific logic is entirely encapsulated within ProviderService.
