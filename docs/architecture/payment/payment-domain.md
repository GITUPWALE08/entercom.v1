# Payment Domain Architecture

## 1. Purpose
The purpose of this document is to define the architectural boundaries, rules, and lifecycle of the Payment domain within the Entercom platform. It establishes the integration requirements with Payment Provider and dictates the handling of financial state changes to ensure system-wide transactional integrity and auditability.

## 2. Scope
* Integration with Payment Provider as the primary payment provider.
* Payment initialization and reference tracking.
* Authoritative Webhook processing and HMAC signature verification.
* Verification of transaction status via server-to-server communication.
* Expiration and failure handling workflows.
* Transactional idempotency for duplicate event notifications.

## 3. Out of Scope
This phase explicitly excludes:
* Refunds, Returns, and Chargebacks.
* Deposits, Installments, and Partial Payments.
* Subscriptions and recurring billing.
* Coupons, Discounts, and Gift Cards.
* Backorders and Reservations.
* Multi-currency support (Defaulting to local currency).
* Notifications and WebSocket broadcasts (Phase 6).

## 4. Definitions
* **Payment Record:** The internal financial aggregate linked 1:1 to an Order.
* **provider_reference:** The unique identifier generated by the system to track the transaction with the provider.
* **Webhook Idempotency:** The mechanism ensuring a single transaction is only processed once, regardless of redundant provider notifications.
* **Canonical Truth:** The rule that the system only updates state based on verified server-to-server data.

## 5. Ownership & Authority
* **Canonical Payment Truth:** The **Webhook is the sole authoritative source of truth**. Backend state MUST NOT be updated based on client-side success signals.
* **Frontend Integrity:** The frontend is NOT a source of truth. Success callbacks are used for UI flow and redirection only.
* **Verification:** The backend is responsible for verifying the HMAC signature of every incoming webhook to ensure authenticity.

## 6. Lifecycle & State Ownership
A Payment record progresses through the following states:
1. **Pending:** Payment initialized; reference shared with Payment Provider/Customer.
2. **Paid:** Terminal state. Verified webhook confirms full settlement. Emits payment.paid event.
3. **Failed:** Non-terminal state. Provider confirms a declined attempt. Order remains `pending_payment` to allow Customer retry.
4. **Cancelled:** Terminal state. Triggered by 24-hour expiration or manual order cancellation.

## 7. Transition Matrix

| Start State | End State | Trigger Source | Rule |
| :--- | :--- | :--- | :--- |
| `None` | `pending` | Order Creation | System initialization. |
| `pending` | `paid` | Webhook (Success) | Verified server-to-server data only. |
| `pending` | `failed` | Webhook (Failed) | Non-terminal; allow recovery. |
| `pending` | `cancelled`| Background Job | Occurs at T+24 hours. |
| `failed` | `pending` | Customer Retry | New provider session; same Order. |
| `failed` | `cancelled`| Background Job | Occurs at T+24 hours. |

## 8. Webhook & Idempotency Rules
* **Duplicate Detection:** The system uses the provider_reference as an idempotency key. Redundant webhooks for terminal states (`paid`/`cancelled`) are acknowledged but ignored.
* **Transactional Integrity:** State updates (Payment Success -> Order Paid -> Inventory Reduced) must occur within a single atomic database transaction.

## 9. Expiration & Failures
* **24-Hour Expiration:** Payment expiry ownership belongs exclusively to a Background Job. Expiry is not user initiated. If a payment is non-terminal for > 24 hours, the system transitions the Payment, Order, and parent Request to `cancelled`.
* **Failure Handling:** A `failed` state allows the Customer to re-initiate payment against the same Order until the expiration window closes. Retries use the SAME Payment record, but a new provider_reference is generated. No new Payment object is created.

## 10. Correlation & Data Requirements
Every Payment record MUST store:
* `order_id`: Linking back to the financial aggregate.
* `request_id`: Linking back to the primary customer aggregate.
* `provider_reference`: The provider-facing identifier.
* `correlation_id`: For system-wide tracing.
* `raw_payload_log`: Secure storage of the verified webhook data.

## 11. Integration Hooks (Events)
The Payment domain emits the following events for downstream consumption (e.g., Phase 6 Notifications):
* `payment.initialized`
* `payment.paid`
* `payment.failed`
* `payment.cancelled`

## 12. Audit Requirements
The system MUST capture:
* Every signature verification attempt result.
* Raw webhook payloads for forensic audit.
* Actor ID (System/Automated for webhooks).

## 13. Completion Criteria
* Webhook processing prevents frontend manipulation of payment states.
* Transactional safety ensures inventory is only reduced when payment is 100% confirmed.
* 1:1 mapping between Order and Payment is strictly enforced.
* 24-hour expiration policy is handled autonomously by the background layer.
