# Booking Domain Glossary

## 1. Purpose
The purpose of this document is to define the canonical terminology for the Booking and Scheduling domain within the Entercom platform. It ensures a ubiquitous language is shared across all architectural, business, and implementation discussions, distinctly separating the concepts of work definition, work assignment, and work scheduling.

## 2. Scope
This glossary defines the foundational nouns and verbs associated with time-bound fulfillment of Requests, the management of technician capacity, and the lifecycle of appointments.

## 3. Out of Scope
* Technical implementation definitions (e.g., database schemas, foreign keys, API endpoints).
* Terminology already defined in the Request Domain Glossary.
* Generalized platform terminology (e.g., User, Role, Audit Log).

## 4. Definitions

* **booking**: A time-bound commitment subordinate to a Request, representing the specific temporal window during which the assigned work will be fulfilled. A Request may exist without a Booking, but a Booking cannot exist without a Request.
* **booking request**: The initial intent expressed by a Customer indicating that a Request requires on-site or scheduled fulfillment.
* **booking creation**: The automated system action that instantiates a subordinate Booking entity immediately following a Technician's explicit acceptance of an Assignment.
* **unscheduled booking**: A Booking entity that has been automatically generated by the system following assignment acceptance, but lacks a definitive start time, end time, and date.
* **scheduled booking**: A Booking entity that has been explicitly assigned a temporal window by a scheduling authority.
* **booking lifecycle**: The state machine progression of a Booking (`unscheduled`, `scheduled`, `in_progress`, `completed`, `cancelled`, `no_show`), which remains entirely subordinate to the parent Request's lifecycle.
* **booking applicability**: The domain rules dictating whether a specific Request workflow progression warrants the creation of a Booking. 
* **booking duration**: The estimated chronological length of the Booking, initially provided by the technician, which dictates the size of the block required in the schedule.
* **booking window**: The specific start and end temporal boundaries allocated on the calendar for a scheduled booking.
* **technician availability**: The aggregate of open temporal slots within a technician's defined working hours, minus any existing scheduled bookings.
* **working hours**: The baseline schedule (days and hours) defined during a technician's onboarding and registration, representing their total potential capacity before bookings are applied.
* **reschedule**: The action of altering the booking window of a previously scheduled booking, subject to strict domain limits.
* **no-show**: An event declared when the required parties (Technician or Customer) fail to be present during the agreed booking window, resulting in immediate cancellation.
* **booking conflict**: A temporal collision where a proposed booking window overlaps with an existing scheduled booking for the same technician.
* **booking owner**: The Customer associated with the parent Request to which the Booking is subordinate.
* **scheduling authority**: The specific actors (strictly Staff) granted the permissions to transition an unscheduled booking into a scheduled booking.
* **booking cancellation**: The termination of a Booking, which inherently cascades to or is driven by the cancellation policy of the parent Request.
* **appointment**: A business term synonymous with a scheduled booking; the formalized agreement of time and place for work fulfillment.
* **fulfillment window**: The practical time block required to complete the Request, encompassing travel, prep, and the core booking duration.

## 5. Rules

### Domain Differentiation
To maintain architectural integrity, the system strictly differentiates between the following concepts:
* **Request** = *What* work is required. (The canonical definition of the customer's need).
* **Assignment** = *Who* performs the work. (The binding of a Technician to a Request).
* **Booking** = *When* the work occurs. (The temporal scheduling of the Assignment).
* **Schedule** = The aggregate calendar view of a Technician's Bookings constrained by their Working Hours.
